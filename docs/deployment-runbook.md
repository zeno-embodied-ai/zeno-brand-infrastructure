# Zeno 数字基础设施部署运行手册

状态：`proposed`

当前许可：Phase 3 仅允许官网与文档站本地实现、静态构建、离线验证和部署文档

禁止事项：当前不得登录生产账户、修改 DNS、创建云资源或部署站点

## 1. 目的与范围

本手册定义 Zeno 官网、文档站、域名/DNS、GitHub Organization、Google Workspace 和监控的标准变更流程。目标是让每项变更有明确负责人、批准、验证、观测与回滚，而不是提供绕过审批的一键生产脚本。

本手册覆盖：

- `zeno-ai.org` 及经批准子域的上线、变更与退役；
- `website/` 和 `documentation/` 的构建与发布；
- `infrastructure/` 中声明式配置的计划、评审和应用；
- GitHub、Workspace 和监控治理基线的分阶段实施。

不覆盖产品机器人固件、实验数据、用户生产数据和付费采购流程。

## 2. 角色与职责

每次变更必须在任务或变更单中映射以下角色；同一人可承担多个低风险角色，但高风险变更应职责分离。

| 角色 | 职责 |
| --- | --- |
| Change Owner | 定义范围、计划、验收标准并协调全过程 |
| Approver | 确认风险、窗口、权限与回滚可接受 |
| Executor | 在批准窗口执行已评审步骤，不临场扩大范围 |
| Verifier | 独立检查用户路径、DNS/TLS、监控和数据完整性 |
| Rollback Owner | 判断并执行回滚，确保旧版本或旧记录可用 |
| Communications | 发送维护、事件和恢复通知 |

DNS、身份、Organization Owner、账单或域名注册变更至少需要平台 Owner 与 Security/Operations 双重批准。

## 3. 变更分级

| 等级 | 示例 | 最低门禁 |
| --- | --- | --- |
| L0 文档 | 架构文档、无执行能力的模板 | PR 评审、本地检查 |
| L1 低风险 | 文案、样式、非关键文档发布 | PR、预览、自动检查、内容 Owner 批准 |
| L2 中风险 | 新子域、网站框架升级、公开工作流权限变化 | 测试环境、Security/Platform 评审、回滚演练、窗口 |
| L3 高风险 | MX/身份、DNS 迁移、Organization Owner、账单、注册商 | 双人批准、带外恢复、维护窗口、逐步人工确认 |

不确定时提升一级。一次变更不要混合无关的 L1 与 L3 项目。

## 4. 安全前提

- 仓库、Issue、PR、命令、日志和截图中不得出现真实密码、API Token、Cookie、私钥、恢复码或付款信息。
- 凭据由经批准的 Secret Manager/CI Secret 在运行时注入，使用短期、单用途、最小权限身份。
- 执行前验证 MFA、备用管理员和带外恢复方式；恢复资料不得写入本仓库。
- Terraform/OpenTofu 状态、平台导出和审计日志可能含敏感数据，应保存在受控后端，不提交 Git。
- 第三方 GitHub Action 固定到不可变提交版本，工作流权限默认只读，Fork PR 不接触生产 Secret。
- 所有输出先评估是否包含成员、邮箱、内部 ID、Token 或 Cookie；无法保证脱敏时停止执行。

## 5. 变更单模板

在 `tasks/` 中为每次实施建立 Markdown 文件，至少包含：

```markdown
# <变更标题>

- 状态：proposed
- 等级：L0/L1/L2/L3
- Change Owner：<角色或公开联系组>
- Approver：<角色>
- Executor：<角色>
- Verifier：<角色>
- 计划窗口：<时区和起止时间>
- 关联 PR：<链接或编号>

## 目标与非目标
## 当前状态（脱敏）
## 前置条件与依赖
## 实施步骤
## 验证与观测
## 回滚触发条件
## 回滚步骤
## 沟通计划
## 执行记录（不得粘贴 Secret）
## 结果与后续事项
```

状态按 `proposed → approved → in-progress → validating → completed` 流转；取消使用 `cancelled`，回滚完成使用 `rolled-back`。

## 6. 通用部署流程

### Gate 0：范围与授权

- [ ] 目标、非目标、影响用户和依赖已写入变更单。
- [ ] 变更等级、Owner、Executor、Verifier 和 Rollback Owner 已确认。
- [ ] 生产实施得到显式批准；没有批准时到此停止。
- [ ] 维护窗口、冻结期、比赛/课程关键时间和沟通计划已检查。
- [ ] 需要的新权限单独申请，未使用共享超级管理员账号。

### Gate 1：只读盘点

仅在获得生产只读访问授权后执行：

- 导出当前 DNS、域名注册、托管绑定、证书、重定向和安全头的脱敏摘要。
- 记录 GitHub 组织策略、仓库列表、Owner/管理员角色和域名验证状态。
- 记录 Workspace 的域名、MX、管理员角色、群组策略和安全基线，不导出用户隐私数据。
- 记录现有监控、告警接收角色、SLO 和状态页依赖。
- 将与 [`domain-matrix.md`](domain-matrix.md) 的差异写入变更单，不直接修正生产。

盘点输出必须脱敏；原始导出保存在受控存储并设置访问和保留策略。

### Gate 2：构建与离线验证

- 在锁定依赖的干净环境中构建 website/documentation 或校验 IaC。
- 运行单元测试、格式检查、链接检查、依赖/许可证和 Secret 扫描。
- IaC 只执行 format、validate、policy check 和 plan；禁止 `apply`。
- 保存构建版本、源码提交、依赖锁文件摘要和检查结果。
- 验证产物不含 `.env`、源码映射中的 Secret、内部 URL 或个人数据。

### Gate 3：预览与验收

- 发布到隔离的临时 URL 或测试环境，不连接生产数据和生产 Cookie。
- 验证中英文页面、移动端、键盘导航、颜色对比、404、重定向和站内搜索。
- 验证产品名称、链接、版本、版权/许可和联系渠道。
- Security 审查响应头、第三方脚本、Cookie、CORS、CSP 报告和依赖风险。
- Content/Product Owner 在预览环境签字；记录已知问题及是否阻断。

### Gate 4：可观测与回滚就绪

- 预先配置并测试 DNS、HTTP、关键页面、TLS 到期和发布失败监控。
- 告警至少有主接收角色和备用接收角色，不只依赖同一故障域。
- 上一个稳定构建保持可重新部署；DNS 旧值和 TTL 已记录。
- 回滚步骤已经在非生产环境演练，预计恢复时间满足业务需要。
- 定义错误率、不可达、证书、登录或邮件异常等量化回滚触发条件。

### Gate 5：生产执行（必须另行授权）

以下步骤是未来执行顺序，不是本阶段授权：

1. Executor 宣布窗口开始，复核批准、范围、身份和目标环境。
2. 检查当前生产状态与盘点快照是否仍一致；漂移则暂停并重新评审。
3. 发布不可变构建或逐项应用已批准配置，不临场加入顺手修改。
4. DNS 变更按单条记录执行，每条记录完成后验证再继续。
5. 身份与邮件变更先验证备用管理员和邮件连续性，最后才收紧旧权限。
6. 每一步将时间、动作、结果和非敏感证据写入变更单。
7. 任一阻断条件触发时停止后续步骤，由 Rollback Owner 决定回滚。

### Gate 6：生产验证

Verifier 从用户视角和独立网络路径检查：

- 根域、`www` 和文档站 DNS 解析符合批准目标；无意外记录。
- HTTP 到 HTTPS、`www` 到根域、旧路径到新路径的状态码与路径保留正确。
- TLS 主机名、证书链和有效期正常；没有混合内容。
- 首页、产品页、Quickstart、搜索、下载和公开联系入口正常。
- 安全头、Cookie、CORS、robots、sitemap 和 canonical 符合设计。
- GitHub 链接、Release/文档版本和开源许可证指向正确权威来源。
- Workspace 邮件变更时，执行经批准的收发验证且不在记录中暴露邮件内容。
- 监控收到健康信号，发布事件与应用日志无异常趋势。

验证窗口至少覆盖 DNS 传播和一段正常使用流量；不要因单一探测成功立即关闭变更。

### Gate 7：关闭与复盘

- 将变更单标记为 `completed` 或 `rolled-back`，记录实际结果和偏差。
- 更新域名矩阵、架构、Owner、资产清单和运行手册。
- 恢复临时降低的 TTL，删除临时权限、预览资源和过期验证记录。
- 观察期结束后关闭窗口；异常或近失误安排无责复盘。
- 不在复盘中粘贴原始 Token、Cookie、用户数据或敏感日志。

## 7. 工作负载专项步骤

### 7.1 官网与文档站

1. 确认技术栈、运行时、依赖锁文件、Node/Python 版本和构建命令已版本化。
2. PR 生成隔离预览；Fork PR 不获得生产 Secret。
3. 产物通过链接、可访问性、SEO 元数据、资源大小和安全检查。
4. 生产发布采用不可变产物；同一提交不得在生产环境临时重新打包。
5. 先发布托管平台 URL 并验证，再绑定/切换自定义域。
6. 保留上一个稳定版本的一键平台回滚能力，但回滚动作仍需授权。

### 7.2 Cloudflare 与 DNS

1. 先确认 Cloudflare 是否为已批准的权威 DNS/CDN；目录名称不代表已采用。
2. 只读导出当前 zone 摘要，与声明式计划比较，人工识别删除和替换操作。
3. 首次导入不得让工具自动接管未知记录；邮件、验证和第三方记录逐一认领 Owner。
4. 代理、缓存、WAF、Bot、防火墙和重写规则分别评审，避免一次切换全部能力。
5. DNSSEC 必须协调注册商 DS 与权威 DNS，先准备恢复路径再启用。
6. 变更后从多个解析器验证权威答案；不得仅依赖控制台显示成功。

### 7.3 GitHub Organization

1. 确认至少两名受保护的 Organization Owner，并使用个人账号、MFA 和恢复流程。
2. 优先使用 Teams 管理仓库权限，避免逐仓库长期直授个人管理员。
3. 配置默认仓库权限、分支保护/规则集、CODEOWNERS、Secret scanning 和 Dependabot 策略。
4. GitHub App 优先于个人访问 Token；记录权限、Owner、安装范围和复审日期。
5. 工作流权限默认只读；发布身份使用 OIDC 短期交换，不保存长期云密钥。
6. 域名验证、SAML/SCIM 等能力按实际套餐和组织需求单独评估，不假设可用。

### 7.4 Google Workspace

1. 保留至少两名受保护的超级管理员；日常操作使用权限更小的委派角色。
2. 先盘点主域、别名域、MX、发信系统、群组、共享盘和恢复渠道。
3. 强制 MFA、管理员告警、第三方 OAuth 应用治理和离职/停用流程。
4. 公开角色地址使用受控群组，配置外部发件、成员可见性、归档与审核策略。
5. SPF/DKIM/DMARC 分阶段实施并观测，邮件认证变更必须有邮件连续性回滚计划。
6. 不把用户目录、群组成员、审计日志或恢复代码导出到本仓库。

### 7.5 监控与状态通告

1. 根据 [`domain-matrix.md`](domain-matrix.md) 为每个 active 服务建立最低探测和 Owner。
2. 监控控制面与被监控站点尽量分离；状态页不得完全依赖同一托管故障域。
3. 告警包含服务、症状、开始时间、严重度和 Runbook 链接，不包含 Secret 或个人数据。
4. 在上线前测试告警送达、升级和解除通知；避免第一次告警发生在真实故障。
5. 对外事件只发布确认事实、影响、缓解和下次更新时间；敏感根因留在受控系统。

## 8. 回滚规则

满足任一条件通常应暂停或回滚：

- 关键入口持续不可达、TLS 无效或重定向循环；
- 错误率、延迟或构建失败超过变更单阈值；
- 登录、邮件收发、GitHub 权限或管理员恢复能力异常；
- 发现 Secret、个人数据、错误授权或第三方接管风险；
- 实际生产状态与批准计划不同，且无法在窗口内安全解释。

通用回滚顺序：停止继续发布 → 保护证据并脱敏 → 恢复上一个不可变构建或批准的旧配置 → 验证核心路径 → 通知相关方 → 持续监控 → 建立后续复盘。不要为掩盖问题删除日志，也不要把敏感证据复制到 Git。

DNS 回滚受缓存和 TTL 影响，恢复旧值不等于立即恢复所有用户；状态通告和持续观测必须覆盖传播窗口。

## 9. 紧急变更

安全事件或大面积故障可缩短常规流程，但不能取消身份确认、双人控制、日志脱敏和事后复盘：

1. 事件负责人声明紧急等级、影响和临时授权范围。
2. 采取最小、可逆的止损动作，不顺带实施计划变更。
3. 每一步由第二人确认并记录时间与结果。
4. 服务稳定后撤销临时权限，24–72 小时内补齐评审和复盘（以组织政策为准）。

若疑似凭据泄露，优先撤销和轮换；仅删除文件或重写 Git 历史不能使已泄露凭据重新安全。

## 10. 当前规划阶段允许的验收

当前 Phase 3 只执行公开技术资料阅读、官网/文档站本地构建和以下检查，不创建云项目、不部署生产站点、不修改 DNS：

```bash
./scripts/check-repository.sh
cd website && npm run check
cd documentation && npm run check
git diff --check
git status --short
```

本阶段完成只代表本地静态官网和文档站通过检查，不代表候选域名可注册、站点已发布或任何生产服务已部署。创建 Cloudflare Pages/Vercel 项目、发布预览/生产部署、绑定域名或修改 DNS 前，必须指定 Owner、完成部署评审，并针对每项受保护操作取得明确授权。
